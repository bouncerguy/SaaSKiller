# Open Source Self-Hosted Business Operating System — Replit Build Prompt

> **CONTEXT**: This project already has calendar functionality built. DO NOT rebuild the calendar. Integrate all new features around the existing calendar module. The calendar is the starting point — everything below extends the project into a full business operating system.

---

## PROJECT OVERVIEW

Build an open-source, self-hosted business operating system for brands and solopreneurs. This replaces all SaaS products (CRM, support desk, invoicing, project management, etc.) with a single self-hosted platform. The user downloads the code, puts it on their own server, and owns their entire infrastructure with zero recurring SaaS fees.

**Target audience**: Small businesses, solopreneurs, agencies, and consultants who want to own their stack.

**Open source**: This will be published on GitHub. Code must be clean, well-documented, and modular.

---

## TECH STACK

- **Frontend**: React with modern responsive design
- **Backend**: Node.js / Express
- **Database**: PostgreSQL (Neon cloud-compatible, but runs anywhere)
- **Real-time**: WebSockets (Socket.io)
- **Deployment**: Replit-ready, but portable to any Node.js host
- **Package manager**: npm

## CRM SYSTEM (/hud/crm)

This must be **as robust as HubSpot**. Two completely separate views with visual differentiation:

### Lead Management (visually distinct — e.g., blue-tinted header/accent)

**Awareness Stage:**
- Track website visitors by IP address
- Gather as much anonymous data as possible (pages visited, time on site, referral source)
- This is viewable but NOT part of the active sales pipeline
- Toggle: show/hide awareness data
- As more data is gathered, attempt to match anonymous visitors to known leads

**Lead Pipeline:**
- Kanban board view with customizable stages
- Create multiple pipelines (one per product/service or custom)
- Default pipeline: Awareness → New Lead → Contacted → Qualified → Proposal → Negotiation → Won/Lost
- Each pipeline stage can have automated sequences

**Lead Sequences (Default: 13 contacts in 30 days):**
- Email sequences (via AI agent + SMTP)
- SMS sequences (via AI agent + Twilio)
- Call reminders/scheduling
- Configure tone, messaging, and context per pipeline
- Each pipeline can have its own SMTP configuration
- Each pipeline can have its own AI agent with different context/SOPs
- Per-pipeline SOP (Standard Operating Procedures) field — text/markdown that gets injected into the AI agent's context window for that pipeline

**Lead Record:**
- Full contact information
- Source tracking (which form, which page, referral)
- All interactions logged with timestamps
- Notes section (timestamped, attributed to user who added)
- Form submission history (linked by email match)
- Activity timeline

### Customer Management (visually distinct — e.g., green-tinted header/accent)

Leads become customers ONLY when they start paying. These are separate database tables.

**Customer Record:**
- Full contact information (name, address, phone, email)
- Searchable by ANY field
- Billing type and payment method (credit card on file via Stripe/PayPal)
- Update credit card / update address functionality
- All products and services they're subscribed to
- Payment history: last payment date, next due date, amount
- **Status indicator (traffic light):**
  - 🟢 Green: Paid up / current
  - 🟡 Yellow: 30+ days past due
  - 🔴 Red: 60+ days past due / collections / retainer hours depleted
- Auto-suspend toggle per service (yes/no — configured at product level)
- Trouble ticket history (viewable from customer record)
- Form submission history (linked by email)
- Complete activity/change history (every action logged: who, when, what)
- Notes section (timestamped, attributed — submit clears the box for next note)
- Password reset initiation (sends email to customer)
- Link to navigate customer to forgot-password page

## SUPPORT / TROUBLE TICKETS (/hud/support)

### Ticket Creation:
- Employees/AI agents can create tickets on customer accounts
- Customers can submit tickets from customer portal
- Incoming support emails (per-product support address) auto-create tickets
- Incoming support texts (Twilio) auto-create tickets
- AI agent auto-categorizes tickets into the correct product/category

### Ticket Kanban Board:
**AI-Handled Board:**
- New → AI Reviewing → Awaiting Customer Response → Resolved/Closed
- AI can: accept ticket, respond, request more info, close ticket
- For informational tickets, AI handles end-to-end

**Human-in-the-Loop Board (separate kanban):**
- AI escalates here when it can't resolve
- When escalated, notify the human(s) responsible (configurable: email, text, or just kanban flag)
- Human can: respond, call customer, kick back to AI to finish, close ticket
- AI puts ticket in human-in-loop category; human resolves and can return to AI

### Ticket Features:
- Full conversation history (threaded)
- Linked to customer record (visible from customer view)
- Status tracking with timestamps
- Priority levels
- Assignment to specific employees or groups
- SLA tracking (optional)

## CRITICAL REQUIREMENTS

1. **Log everything.** Every action, every change, every view — timestamped and attributed.
2. **Help on everything.** Every feature gets a `?` icon with explanation. Build help content as you build features.
3. **Separate leads from customers.** Different tables, different visual treatment, different workflows.
4. **Permission-gate everything.** Every feature checks group + individual permissions before rendering.
5. **Per-pipeline configuration.** Each pipeline gets its own SMTP, AI agent, SOPs, and sequences.
6. **Per-product configuration.** Each product gets its own support email, AI agent, SOPs, and settings.
7. **Never expose passwords.** Employees can initiate resets but never see credentials.
8. **Modern security.** CSRF protection, rate limiting, input sanitization, parameterized queries, httpOnly cookies, HTTPS enforcement.
9. **The calendar already exists.** Do not rebuild it. Integrate it into the /hud layout and connect it to the user/permissions system.
10. **This is open source.** Write clean, documented, modular code. Others will read and contribute to this.
